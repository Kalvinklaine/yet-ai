# 036 Browser-first Project Isolation

## Status

Approved architecture for the next implementation milestone. This record freezes contracts and rollout rules; it does not claim that project registration, project-scoped routes, storage migration, or the project hub are implemented.

## Decision

Yet AI will make an explicitly registered project the authority boundary for browser navigation and project data. In v1, one project is exactly one canonical directory root. The engine owns the private root registry, generates opaque project IDs, resolves every project-scoped request to an immutable `ProjectContext`, and keeps project data in project-ID-keyed Yet AI config and cache namespaces.

This is a browser-first milestone. `/projects` is the normal browser landing page and primary project hub. IDE project binding is deferred.

The design remains local-first BYOK. Core project workflows, project storage, provider setup, and direct configured-provider or local-runtime access require no hosted Yet AI backend, Yet AI account, managed model gateway, product credit balance, or cloud workspace.

## Terminology and identity

| Term | Meaning | Exposure |
| --- | --- | --- |
| Project | One registered canonical directory root plus engine-owned metadata. | Safe metadata may be shown locally. |
| Canonical root | The absolute, normalized filesystem identity returned by successful canonicalization at registration or revalidation. | Private registry material. |
| `projectId` | Durable opaque identifier generated by the engine. | Safe for local routes and UI metadata. |
| Display label | Sanitized user-facing project name; it is not an identifier and need not be unique. | Safe for local UI metadata. |
| `ProjectContext` | Immutable request-scoped resolution of an active project and its scoped storage. | Engine internal; only sanitized status is returned. |
| Legacy unscoped data | Data created before project scoping and retained outside every registered project. | Exposed only through an explicitly labeled compatibility entry. |

### Project ID contract

The engine generates each ID as `prj_` followed by the unpadded 22-character base64url encoding of 16 cryptographically random bytes. The complete textual form is therefore 26 ASCII characters, path-safe, and carries 128 bits of random entropy. Clients cannot choose, derive, parse for meaning, or recycle an ID. The engine retries on the exceptionally unlikely registry collision and commits the ID only with the registry record.

Canonical roots never become identifiers. A rename, duplicate label, moved root, or changed filesystem spelling does not alter a project's ID.

### Public and private fields

`projectId` and the sanitized display label are public local UI metadata. The canonical root, filesystem identity details, discovery handles, storage paths, and raw filesystem errors are private engine material. Canonical roots may be shown only in an explicit local project-management confirmation or repair flow where the user needs to identify a directory. They must not appear in durable URLs, routine project summaries, model prompts, exports, normal logs, activity summaries, error bodies, or the normal project hub.

## Trust and authority boundaries

The existing local bearer or browser-session authentication remains user-level authority. Project ownership is enforced independently by route resolution and scoped storage:

1. Authenticate the request using the existing local user/session boundary.
2. Parse and validate the route's `projectId`.
3. Resolve the ID from the engine registry and apply archived/availability policy.
4. Construct one immutable `ProjectContext` for that request or stream.
5. Open only storage derived from that context.

Authentication alone never selects a project. A project URL contains no bearer token, browser-session secret, canonical root, or discovery handle. There is never a mutable process-global “current project.” Concurrent requests, tabs, and streams can safely resolve different project contexts in one engine process.

A conceptual `ProjectContext` contains the project ID, registry revision, sanitized label, availability/archive state, private canonical root reference, and resolved project config/cache roots. It is immutable after route resolution. Long-lived streams retain the context and generation with which they opened; they do not follow later client navigation.

## Registration and canonicalization

### Inputs

Project creation accepts an authenticated engine-issued directory handle and a sanitized optional display label. The CLI fallback accepts a local root argument through its own authenticated/local invocation boundary. Neither API accepts a browser-submitted absolute path as a normal registration input.

The engine validates label length and characters, resolves the handle, confirms that the target is a readable directory, canonicalizes it, checks policy, reuses an existing project for the same canonical root or otherwise generates the ID, and atomically records a new project. Registration does not scan file contents, index the directory, infer hidden context, or call a provider.

### Symlinks and filesystem identity

Registration follows the platform filesystem's canonicalization operation once and stores the resulting canonical directory root. A symlink chosen by the user is therefore registered as its resolved target, not as a second alias. On Unix platforms the engine also stores the canonical directory's bounded numeric device and inode fields as private filesystem identity. Duplicate registration and every availability or open check require both the canonical root and this identity to match, so removing a directory and recreating another directory at the same path cannot capture its project ID. Registry v1 and v2 records have no independently trustworthy identity proof, so migration never derives identity from the current filesystem. Registry v3 records with a concrete identity migrate as bound, while a null identity is ambiguous because it may represent either an older unsupported platform or missing evidence; it therefore migrates as unbound rather than unsupported. Every unbound record remains unavailable until a later explicit repair or rebind establishes identity, even when a readable directory currently exists at the stored path, and registering that same root cannot silently reuse the record. Registry v4 distinguishes that fail-closed unbound state from the explicitly unsupported identity capability used by new registrations on non-Unix platforms.

On non-Unix platforms this milestone does not claim a stable filesystem identity primitive. The stored identity is absent and the shared predicate conservatively retains canonical-path, directory-type, and readability checks. Tests cover that fallback separately from the stronger Unix replacement detection contract.

The engine rejects a missing target, non-directory target, canonicalization loop, broken symlink, inaccessible target, or target that escapes the active browser-discovery root. It also rejects unsafe symlinks in engine-owned registry and project storage paths. Files and symlinks inside a registered project are not traversed merely by registration.

Canonicalization is repeated during explicit availability checks and before a root-sensitive operation. If the stored root is now a symlink to another location, resolves to a different directory, or cannot be resolved, the project becomes unavailable rather than silently changing identity. Repair or rebind requires an explicit user action that proves the intended directory and preserves the existing `projectId` only when policy permits it.

### Outputs and duplicate rules

Successful registration returns safe project metadata: `projectId`, sanitized label, archive state, availability/readiness, `revision`, and safe timestamps or bounded activity summary when present. A newly registered project has `lastOpenedAt: null`; only a later successful project open sets that timestamp. `createdAt` is always present. Normal create/list/open responses do not return the canonical root.

Display labels are not unique. Duplicate labels are allowed and the UI disambiguates rows with safe status and recent-activity metadata, never by exposing roots in the normal hub. Registering the same canonical root again is a successful idempotent operation: it returns the existing safe summary and consumes the discovery handle without changing the display label, revision, `createdAt`, or `lastOpenedAt`. It does not disclose the root. Distinct canonical roots always create distinct projects.

## Browser directory discovery

Browser registration uses a bounded, authenticated, engine-owned discovery session:

1. The user chooses **Add local project**.
2. The GUI starts a short-lived discovery session through `/v1/project-browser/sessions`.
3. The engine roots that session at the user's home directory and returns only opaque handles plus sanitized entry labels and kinds.
4. The GUI navigates directory entries by presenting an opaque parent handle; it never constructs or submits raw paths.
5. The user explicitly selects one directory and confirms registration by sending `{directorySessionId,directoryHandle,displayName}` to `POST /v1/projects`.
6. That one operation resolves the handle inside the same authenticated session, canonicalizes it, rechecks containment and policy, atomically records a new project or resolves the existing same-root project, and consumes the handle only after either outcome succeeds.

Sessions and handles are random, user-session-bound, short-lived, bounded in entry count, page size, navigation depth, and lifetime. A handle remains valid until a successful registration consumes it, the session expires or is cancelled, authentication changes, or the engine restarts. Validation failures do not create a project and do not consume the handle, so the caller may correct a retryable request and retry while the session remains valid. Failures that prove the session or handle invalid return `discovery_expired`; policy and filesystem failures preserve the handle only while it remains safe and valid. Discovery lists directories only. It performs no recursive discovery, file-content read, background scan, indexing, hidden project inference, or provider call. Hidden directories are omitted by default; any future reveal control requires an explicit product decision.

The browse root is the user's canonical home directory. Roots outside it are registered with the CLI fallback. The API does not widen discovery authority in response to `..`, symlinks, crafted handles, alternate path spellings, or client-supplied roots.

## Routes

All routes are same-origin local product routes unless explicitly documented otherwise. Unknown, malformed, missing, archived, or unavailable project IDs fail closed without consulting unscoped storage.

### Browser routes

| Method | Route | Purpose |
| --- | --- | --- |
| `GET` | `/` | Redirect or history-replace to `/projects`. |
| `GET` | `/projects` | Default browser home and primary project hub. |
| `GET` | `/projects/legacy` | Explicit legacy unscoped-data entry when such data exists. |
| `GET` | `/p/:projectId/` | Project shell; redirects to the project's default page. |
| `GET` | `/p/:projectId/chat` | Project conversation list or default chat. |
| `GET` | `/p/:projectId/chat/:chatId` | One project-owned conversation. |
| `GET` | `/p/:projectId/memory` | Project-curated memory. |
| `GET` | `/p/:projectId/agent` | Project agent/progress surface within existing authority limits. |
| `GET` | `/settings` | Global provider, runtime, demo, and preference settings. |

Back and forward navigation restore URL-addressed project pages. They never restore a hidden mutable current project. An archived, missing, or unavailable project URL renders a safe recovery state with a route back to `/projects`.

### Global control API

| Method | Route | Purpose |
| --- | --- | --- |
| `GET` | `/v1/projects` | List safe project summaries, including explicitly requested archived entries. |
| `POST` | `/v1/projects` | Register a project from an engine discovery handle. |
| `GET` | `/v1/projects/:projectId` | Read one safe project summary. |
| `PATCH` | `/v1/projects/:projectId` | Rename or update permitted safe metadata. |
| `POST` | `/v1/projects/:projectId/archive` | Archive a project without deleting its data. |
| `POST` | `/v1/projects/:projectId/restore` | Restore an archived project. |
| `POST` | `/v1/projects/:projectId/revalidate` | Explicitly recheck availability for repair UX. |
| `POST` | `/v1/project-browser/sessions` | Start bounded home-rooted directory discovery. |
| `POST` | `/v1/project-browser/sessions/:sessionId/list` | List bounded child directory handles from a body containing only `directoryHandle`; session identity is path-owned. |
| `DELETE` | `/v1/project-browser/sessions/:sessionId` | Cancel discovery and invalidate its handles. |

The concrete discovery response schemas may be refined before implementation, but they must preserve opaque handles, authentication, bounds, containment, and no raw-path normal responses. There is no project hard-delete route in v1.

### Project data API

| Method family | Route family | Scope |
| --- | --- | --- |
| Existing chat methods | `/p/:projectId/v1/chats...` | Chats, commands, history, and SSE are owned by exactly one project. |
| Existing memory methods | `/p/:projectId/v1/project-memory...` | Curated memory is owned by exactly one project. |
| Read-only progress methods | `/p/:projectId/v1/agent-progress...` | Progress and controlled-run metadata are read only from that project scope. |
| Future project methods | `/p/:projectId/v1/...` | Must resolve `ProjectContext` before storage or work begins. |

A nested resource ID is insufficient authority by itself. For example, a chat ID found under project A must return a sanitized not-found or ownership error when requested under project B.

### Legacy compatibility routes

| Route family | Rollout behavior |
| --- | --- |
| `/v1/chats...` | During compatibility rollout, accesses only isolated legacy unscoped chat data and is never an alias to a selected project. It is deprecated and later disabled. |
| `/v1/project-memory...` | During compatibility rollout, accesses only isolated legacy unscoped memory and is never guessed into a project. It is deprecated and later disabled. |
| `/v1/agent-progress...` | During compatibility rollout, exposes only isolated legacy unscoped progress according to its existing read-only policy. It is deprecated and later disabled. |
| `/projects/legacy` | Explicit GUI entry for available legacy unscoped data; absence of legacy data may return the project hub or a safe empty state. |

Global endpoints such as providers, provider authentication, models, capabilities, runtime authentication, demo mode, and global preferences remain under `/v1/...`; they do not acquire a project prefix merely because the GUI is displaying a project.

## Storage model

The registry is engine-owned private configuration under the Yet AI config namespace. Project data is keyed only by validated project IDs:

```text
<yet-ai-config>/
  projects/
    <projectId>/
      chats/...
      project-memory/...
      controlled-runs/...

<yet-ai-cache>/
  projects/
    <projectId>/
      agent-progress/...
      indexes/...
      temporary/...
```

Exact child filenames remain implementation decisions of their owning subsystems. The fixed contract is `projects/<projectId>/...` beneath the existing `yet-ai` config/cache roots from `product/identity.json`. Path construction never uses a label, canonical root, client-provided segment, or resource ID that has not passed its own validation.

Providers, provider credentials, runtime/browser-session authentication, demo mode, and global preferences remain global. They are not copied into project directories. Project chat, curated memory, agent progress, controlled-run state, and future project-specific indexes are project-scoped. V1 does not place this scoped operational data in the registered repository or use `.yet-ai` as the authoritative browser-project store.

Registry and scoped writes use atomic replacement and private permissions where supported, reject unsafe engine-storage symlinks, and serialize conflicting mutations. Each persisted project summary carries a per-project `revision` encoded as a canonical unsigned decimal string from `1` through 20 digits, without leading zeroes. Registration starts at revision `1`, and v1 registry migration maps every entry to revision `1`; revision `0` is never emitted or accepted as an expected revision. Rename, archive, restore, and successful open require `expectedRevision`; the mutation succeeds only when it exactly matches the current revision and returns the summary or lifecycle response with the revision incremented by exactly one. A mismatch or an unrepresentable increment makes no change and returns `invalid_request` with a bounded non-sensitive conflict message.

All registry reads and mutations take a bounded OS-backed lock on a private `registry.lock` file beside the registry. The synchronous transaction revalidates the complete trusted config, cache, and registry-directory chains immediately before lock creation/open and again before registry use. Lock opening uses no-follow where supported and verifies that the opened handle still identifies the same regular file as the path both before and after acquiring the exclusive lock. If the path was replaced, the stale handle is closed and acquisition retries within the original bounded deadline; registry use never proceeds while the locked handle and path identify different files. Unix verification compares device and inode, Windows verification compares volume serial number and file index, and platforms without a verifiable handle/path identity fail closed. A mutation reloads and validates the latest registry while holding that lock, applies revision and root checks, and atomically replaces the registry. Reads also reload under the lock, so CLI changes become visible to a running engine without restart; process-local ordering uses no registry cache. Lock timeout, unsafe lock-file type, and lock I/O failures map to sanitized storage-unavailable errors. The operating system releases stale locks when a process exits; PID files and lock stealing are not used. When one or more missing private registry ancestors are first created, each new directory receives private permissions before its already-existing parent is synced, including syncing the config directory after creating `projects`. After atomic rename, the project registry directory is synced. Directory sync responses that explicitly report unsupported, invalid, or permission-denied are tolerated; every other sync failure is surfaced as storage unavailable. A post-rename failure may therefore report unavailable even though the replacement is visible, while an ancestor-creation failure stops before lock or registry use.

## Project hub information architecture

`/projects` uses the Yet AI React GUI and product design system. It is not an independent server-rendered admin template or a hidden developer picker. It is the normal browser home.

The page has:

- the Yet AI product header, a clear Projects title, and global Settings access;
- a compact responsive list or table of projects;
- **Add local project** as the primary creation action;
- row-level Open, rename, and archive actions;
- an archived view with restore actions;
- an explicitly labeled legacy-data entry when legacy unscoped data exists;
- loading, empty, partial-readiness, error, unavailable, and archived states;
- keyboard-accessible actions and back/forward-safe links.

## Project switching and concurrent clients

A project switch is a navigation and lifecycle boundary, not a mutation of global engine state. Before committing the new project generation, the GUI must:

1. abort project-bound fetches and pending mutations;
2. close project SSE or other streams;
3. rotate a monotonically changing or random project-generation token;
4. clear chat, memory, progress, controlled-run, selection, draft, error, and transient context state owned by the old project;
5. build clients and subscriptions from the new route's `projectId`;
6. reject every late response, event, timeout, retry, or host result whose project ID or generation no longer matches.

Per-request cancellation is best effort; late-result rejection is mandatory. A global Settings request need not be canceled because it is global, but it cannot repopulate project-bound state.

Multiple tabs may open the same or different projects. Each tab has its own generation and route state; the engine resolves each request independently. Registry mutations use the `expectedRevision` checks above; a stale mutation returns `invalid_request` with a generic conflict message because the frozen public taxonomy has no separate conflict category. Archive/restore events may be observed through explicit refresh or a future bounded registry notification, but no tab silently adopts another tab's current project. A newly archived project stops accepting new project-data mutations; already open streams close or terminate with a sanitized project-state event at the next policy boundary.

## Availability, archive, and repair states

| State | Open behavior | Mutations | Hub behavior |
| --- | --- | --- | --- |
| Ready | Open project shell. | Allowed by endpoint policy. | Normal row. |
| Missing or moved | Show safe unavailable/repair state. | Project-data mutations denied. | Retain row with revalidate/repair guidance. |
| Permission denied | Show safe unavailable state. | Denied. | Retain row without raw OS error. |
| Archived | Do not open normal project pages. | Denied except restore and permitted metadata. | Hidden from active list by default; visible in archived view. |
| Legacy unscoped | Open only explicit compatibility surface. | Existing compatibility policy only. | Separate labeled entry, never a project row. |

A moved directory is never discovered in the background. The user explicitly invokes repair/rebind through bounded discovery or CLI. Repair does not merge two project records, steal a root already owned by another project, or infer identity from label or contents.

Archive is reversible registry state. It does not delete the canonical registry record or project data. Restore revalidates the root and may produce an unavailable restored project if the root is still missing. V1 provides no hard delete, data purge, or repository mutation.

## Error contract

Errors use stable categories, suitable HTTP status codes, and short sanitized messages. They do not echo submitted IDs, handles, roots, storage paths, raw OS errors, parser dumps, tokens, or directory contents.

| Category | Meaning |
| --- | --- |
| `invalid_request` | Malformed or unsafe input, stale revision, unknown field, or exceeded bound. |
| `not_found` | No visible project or nested project resource exists for the scoped request. |
| `archived` | The operation is unavailable while the project is archived. |
| `root_missing` | The registered root is missing, moved, inaccessible, or has changed identity. |
| `discovery_expired` | The discovery session or handle expired, was consumed, or is invalid. |
| `outside_allowed_root` | The selected directory does not remain beneath the approved discovery root. |
| `unsafe_filesystem` | Filesystem permissions, type, canonicalization, or symlink policy denied the operation. |
| `storage_unavailable` | Registry or scoped storage cannot be safely read or written. |

This eight-name set is the complete public project-control taxonomy. Authentication transport failures remain governed by the existing authentication contract rather than introducing another project error category. Internal duplicate, nested-resource, legacy-boundary, revision-conflict, and filesystem details map to the categories above and never create parallel public names.

Callers may retry only categories documented as transient. The GUI maps categories to bounded recovery actions such as retry, return to Projects, restore, revalidate, choose another directory, or use CLI registration. It never renders raw engine or filesystem errors.

## Rollout and migration

Rollout is additive and reversible by wave:

1. Add registry and project contract support without moving existing data. Keep all legacy unscoped data untouched and labeled.
2. Add authenticated global project controls, discovery, and project-scoped route resolution. Project routes write only new project namespaces.
3. Add project-scoped chat, memory, progress, and controlled-run storage. No code falls back from a missing scoped record to unscoped data.
4. Ship the React `/projects` hub and project shell, then make `/` redirect to `/projects`.
5. Expose legacy data only through `/projects/legacy` and compatibility APIs. Offer explicit copy/import only in a separately reviewed migration flow; never infer a project from paths, labels, recent use, or content.
6. After compatibility evidence and user communication, disable deprecated unscoped data APIs without deleting their files.

Migration operations, if later implemented, must be explicit, previewable, project-targeted, idempotent, and interruption-safe. They copy or move only after validating source generation, target `ProjectContext`, resource ownership, limits, and atomic commit behavior. A failed migration leaves the source intact and does not create a partially visible target resource. Successful migration records safe non-secret audit metadata and never embeds a canonical root in migrated content.

### Rollback

Before the default-route cutover, rollback disables project UI entry points and scoped writes while leaving the new registry and project namespaces intact for retry. After scoped data exists, rollback may restore the old landing route only as a temporary compatibility mode; it must not flatten scoped data into unscoped storage, guess a current project, alias unscoped endpoints to one project, reuse project IDs for another root, or delete registry/data.

Every rollout wave must be readable by the previous deployed schema version or guarded by an explicit version check that fails safely. Registry migrations use backups or atomic versioned replacement. If a schema downgrade cannot safely read new records, startup exposes a sanitized blocked state and recovery instructions rather than rewriting or discarding them.

## Non-goals

V1 does not add:

- a daemon, per-project process, worker, or background session service;
- multi-root projects;
- background directory discovery, project inference, workspace scanning, or indexing;
- hidden context collection or automatic prompt attachment;
- cloud sync or a hosted project service;
- IDE workspace binding or automatic IDE project registration;
- broader agent, shell, git, tool, provider-tool, file-read, or mutation authority;
- project worktrees or task-board ownership;
- hard deletion or guaranteed purge of project data;
- automatic migration or guessed assignment of legacy data;
- raw project logs, daemon logs, process consoles, live log viewers, or daemon controls in the project hub;
- tokens or canonical roots in project URLs;
- copied third-party source, UI, assets, names, documentation, or tracked references.

## Consequences

Project IDs become the durable public local routing key while roots remain replaceable private registry material. Browser navigation is deterministic and shareable on the same authenticated local runtime without leaking filesystem paths. Request-scoped context prevents one tab's selection from changing another request's authority. Project-scoped config/cache storage isolates data without writing operational state into repositories.

The cost is an explicit registry, availability/repair UX, migration compatibility lane, strict GUI invalidation, and additional route ownership checks. Those are required safety boundaries, not optional polish. A project picker that casually remembers a global path would be shorter code, but also a tiny cross-project data catapult. We decline the catapult.
